Part 8: Understanding Firewall Traffic Filtering

How Windows Firewall Works

Through this lab, I learned that Windows Firewall operates as a stateful packet filter that inspects network traffic based on defined rules.

Packet Filtering Process

Network Packet Arrives â†“ Firewall Examines Packet Headers: - Source IP Address - Destination IP Address - Protocol (TCP/UDP/ICMP) - Source Port - Destination Port â†“ Compare Against Firewall Rules (Sequential Order) â†“ First Matching Rule Found â†“ Execute Action: ALLOW or BLOCK â†“ If No Match â†’ Apply Default Policy

Rule Processing Order

Windows Firewall processes rules in this priority:

  1. Windows Service Hardening Rules - System protection rules
  2. Connection Security Rules - IPsec authentication rules
  3. Authenticated Bypass Rules - Rules for authenticated traffic
  4. Block Rules - Explicitly blocking traffic
  5. Allow Rules - Explicitly allowing traffic
  6. Default Rules - Built-in Windows rules
  7. Default Profile Behavior - Block inbound, Allow outbound
Important: Block rules always take precedence over Allow rules when both match the same traffic.

Stateful Inspection

Windows Firewall uses stateful inspection, which means:

  • Tracks Connection State: Maintains a table of active connections
  • Automatic Return Traffic: If I initiate an outbound connection (e.g., browsing a website), the firewall automatically allows the return traffic without needing an explicit inbound rule
  • Connection States:
    • NEW - First packet of a new connection
    • ESTABLISHED - Packet belonging to existing connection
    • RELATED - Packet related to existing connection
Example from Lab: When I tested port 23 with Test-NetConnection, the outbound request was allowed by default policy, but no return traffic was possible because port 23 was blocked inbound.

Traffic Direction

  • Inbound: Traffic coming TO my computer from the network
  • Outbound: Traffic going FROM my computer to the network

Lab Observation:

  • Blocked inbound Telnet (port 23) â†’ External connections to my port 23 rejected
  • Default policy allows outbound â†’ I could still initiate connections to other Telnet servers

Firewall Profiles

Windows Firewall maintains three separate profiles:

  1. Domain Profile: Active when connected to corporate domain network
  2. Private Profile: Active on home/work trusted networks
  3. Public Profile: Active on untrusted public networks

Each profile can have different rules. In this lab, I applied rules to "Any" profile (all three).

Why Blocking Port 23 (Telnet) Matters

Telnet transmits data in plaintext including passwords, making it insecure. Blocking it prevents:

  • Unauthorized access attempts
  • Password sniffing attacks
  • Exploitation of vulnerable Telnet services

Modern alternative: SSH (port 22) with encryption

Real-World Application

Firewalls filter traffic by:

  • Preventing unauthorized access to services
  • Blocking malicious traffic patterns
  • Allowing legitimate applications to communicate
  • Protecting against network-based attacks
  • Enforcing security policies

Observations and Learning Outcomes

Key Takeaways

  1. Rule Specificity: Firewall rules require precise configuration (protocol, port, direction, action) to work correctly.
  2. Testing is Critical: Always test rules after creation to ensure they function as intended. Used multiple methods (Test-NetConnection, Telnet, netstat) for verification.
  3. Default Policies Matter: Windows Firewall's default "block inbound, allow outbound" policy provides good baseline security while maintaining usability.
  4. Order of Operations: Understanding that Block rules override Allow rules is crucial when troubleshooting connectivity issues.
  5. Stateful Benefits: Stateful inspection simplifies rule management by automatically handling return traffic for established connections.

Challenges Encountered

Challenge 1:

Initially confused about why outbound Telnet worked even with port 23 blocked.

Solution: Realized the block rule only affects INBOUND traffic. Outbound connections to other Telnet servers are still possible unless explicitly blocked.

Challenge 2:

Test-NetConnection returned "False" even after creating Allow rule for SSH.

Solution: The firewall rule allows traffic, but connection still fails if no SSH server is listening. Rule is working correctly; just no service to connect to.

Security Implications

  • Port 23 (Telnet): Should remain blocked in production environments due to security vulnerabilities
  • Port 22 (SSH): Safer alternative to Telnet when remote access is needed
  • Principle of Least Privilege: Only open necessary ports; keep unused services blocked
  • Regular Audits: Firewall rules should be reviewed periodically to remove unnecessary exceptions
